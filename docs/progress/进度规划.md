# 毕业设计进度规划文档

## 课题名称
**基于 ROS2 的工业机器人运动控制系统架构设计与仿真验证**

## 学生信息
- 专业：计算机科学与技术
- 学历：本科
- 主要编程语言：C++
- 技术方向：软件开发 / 运动控制

# 阶段 1：开发环境搭建与最小系统验证（详细）

## 阶段目标
本阶段的目标是**完成毕业设计所需开发与仿真环境的搭建，并验证 ROS2 与工业机器人仿真系统的基本可用性**。  
通过运行官方或成熟示例，使机器人模型在仿真环境中成功运动，为后续系统架构实现和功能开发提供稳定基础。

本阶段重点在于：
- 熟悉 ROS2 与仿真平台
- 排除环境与配置问题
- 验证工具链的可行性  
**不涉及系统架构设计与功能创新**

---

## 主要任务

- 操作系统选择：
  - Ubuntu 22.04 LTS
- ROS2 版本选择：
  - ROS2 Humble Hawksbill
- 确认系统满足：
  - C++ 编译环境正常
  - 网络与软件源配置正常

**目标：**
- ROS2 能正常启动
- 基础命令可正常使用

---

### 1.2 ROS2 基础功能验证
- 验证 ROS2 基本组件是否可用：
  - `ros2 node list`
  - `ros2 topic list`
  - `ros2 run` / `ros2 launch`
- 运行 ROS2 官方示例：
  - Publisher / Subscriber 示例
  - 定时器与回调机制示例

**目的：**
- 熟悉 ROS2 节点运行方式
- 理解话题通信与消息发布流程
- 确保 ROS2 运行环境稳定

---

### 1.3 Gazebo 仿真平台搭建
- 安装并配置 Gazebo 仿真环境
- 验证 Gazebo 能独立运行
- 验证 Gazebo 与 ROS2 的联动能力

**验证方式：**
- 启动 Gazebo 空仿真世界
- 确认 Gazebo 窗口正常显示
- 确认 Gazebo 与 ROS2 通信插件加载正常

---

### 1.4 工业机器人模型加载与验证
- 选用 UR5 工业机器人模型作为仿真对象
- 加载 UR5 至 Gazebo 仿真环境
- 验证机器人模型：
  - 结构显示正确
  - 关节数量与自由度正确
  - 关节状态话题可正常获取

**注意事项：**
- 本阶段不修改模型参数
- 不进行自定义 URDF 建模
- 仅验证模型的可用性与稳定性

---

### 1.5 最小运动控制验证（关键）
- 使用已有示例或简单方式控制机器人运动：
  - 通过 ROS2 topic 发布关节指令
  - 或使用现成 demo 控制节点
- 验证以下内容：
  - 机器人能够接收控制指令
  - 关节角发生变化
  - 机器人在仿真环境中产生可见运动

**说明：**
- 运动精度不作为评价标准
- 控制方式可以是最简单形式

# 阶段 2：系统总体架构实现

## 阶段目标
本阶段的目标是**将前期确定的系统方案转化为实际可运行的 ROS2 工程架构**，完成工程目录设计、功能包划分、通信链路设计、接口规范确定以及基础节点实现，为后续运动控制闭环开发打下稳定基础。

本阶段重点在于：
- 完成系统总体架构的工程落地
- 确定工程目录结构与各模块职责边界
- 确定 topic、action、service、msg 的组织方式与存放位置
- 建立统一的参数管理、启动方式与文档规范
- 验证系统各节点之间的通信链路正确性与可追踪性

**本阶段不以复杂运动控制效果为目标，重点在于“工程骨架搭建完成、接口规范冻结、架构链路可运行”。**

## 主要任务

### 2.1 工程总体目录设计与工作空间搭建
在已完成环境验证的基础上，建立毕业设计正式 ROS2 工作空间，并确定后续开发统一使用的工程目录结构。

工程目录如下：

```bash
ros2_ws/
├── src/
│   ├── robot_motion_msgs/
│   ├── motion_api_pkg/
│   ├── planner_pkg/
│   ├── controller_pkg/
│   ├── robot_interface_pkg/
│   ├── system_manager_pkg/
│   └── bringup_pkg/
│
├── build/
├── install/
├── log/
│
├── docs/
│   ├── architecture/
│   ├── interfaces/
│   ├── progress/
│   └── experiments/
│
├── bags/
├── scripts/
└── README.md
```

#### 各目录作用说明

##### （1）`src/`

存放全部 ROS2 功能包源码，是系统核心开发目录。

##### （2）`robot_motion_msgs/`

存放本课题自定义接口定义，包括：

- 自定义消息 `msg/`
- 动作接口 `action/`
- 服务接口 `srv/`

该包作为公共接口层，供其余功能包共同依赖。

##### （3）`motion_api_pkg/`

任务入口层，对外接收运动任务请求，负责创建任务、生成 `task_id`、发起任务执行请求。

##### （4）`planner_pkg/`

运动规划层，负责根据目标关节角生成轨迹数据，后续可扩展不同规划策略。

##### （5）`controller_pkg/`

控制执行层，负责接收规划结果并按固定频率生成控制指令。

##### （6）`robot_interface_pkg/`

机器人接口适配层，负责屏蔽底层控制器差异，将上层控制输出适配为 Gazebo / ros2_control 所需接口。

##### （7）`system_manager_pkg/`

系统管理层，负责状态机、任务生命周期、超时与异常管理。

##### （8）`bringup_pkg/`

统一启动包，存放 launch 文件、统一参数文件和 RViz 配置。

##### （9）`docs/`

存放架构设计文档、接口文档、阶段总结和实验记录，为论文撰写与答辩提供依据。

##### （10）`bags/`

存放 ros2 bag 实验数据，用于后续阶段的数据分析与结果对比。

##### （11）`scripts/`

存放辅助脚本，例如一键录 bag、检查 topic、运行演示脚本等。

### 2.3 通信接口设计与文件位置确定

本阶段需完整确定系统使用的通信机制，包括 action、topic 与 service 的命名、作用及文件存放位置。

#### 2.3.1 Action 接口设计

系统对外运动任务入口采用 Action 机制，以提高任务级控制的工程性、可反馈性与可取消性。

##### Action 名称

```
/move_joints
```

##### 文件存放位置

```
robot_motion_msgs/action/MoveJoints.action
```

##### 作用

- 接收目标关节角运动任务
- 支持执行反馈
- 支持取消任务
- 支持返回执行结果

##### 建议字段

```
# Goal
string task_name
string[] joint_names
float64[] target_positions
float64 speed_scale
float64 timeout_sec
---
# Result
bool success
string task_id
string message
float64 final_error
---
# Feedback
string task_id
string current_state
float64 progress
float64 current_error
```

------

#### 2.3.2 Topic 设计

系统内部模块通信采用 topic 进行解耦。

##### （1）`/planned_traj`

- 作用：规划节点输出轨迹
- 发布者：`planner_node`
- 订阅者：`controller_node`

##### （2）`/joint_cmd`

- 作用：控制节点输出关节控制命令
- 发布者：`controller_node`
- 订阅者：`robot_interface_node`

##### （3）`/joint_states`

- 作用：机器人反馈当前关节状态
- 发布者：仿真控制器 / 机器人控制器
- 订阅者：`controller_node`、`system_manager_node`

##### （4）`/system_state`

- 作用：广播系统状态、任务编号与异常信息
- 发布者：`system_manager_node`
- 订阅者：系统其他节点、可视化工具

##### （5）`/task_event`（可选）

- 作用：广播任务关键事件
- 发布者：各模块节点
- 订阅者：`system_manager_node` 或日志系统

### 2.7 系统通信链路设计

本阶段应完整确定系统主链路与辅助链路。

#### 主链路

```
用户/上层请求
   ↓
Motion API Node
   ↓  (MoveJoints Action Goal)
Planner Node
   ↓  (/planned_traj)
Controller Node
   ↓  (/joint_cmd)
Robot Interface Node
   ↓
Gazebo / ros2_control / UR5
   ↓  (/joint_states)
Controller Node / System Manager Node
```

#### 辅助链路

- `System Manager Node` 监听任务状态与系统运行事件
- 发布 `/system_state`
- 维护当前 `task_id` 与系统状态流转
- 对异常、超时、重置等事件进行统一管理

- `task_id` 能在日志中从上游传递到下游

- 文档与配置文件位置固定

说明：
